
## Iteration 7 - US-025: Port TUI core
- What was done:
  - Created `crates/tau-tui` workspace member.
  - Ported `utils.rs` (ANSI string manipulation), `component.rs` (Component/Container), and `terminal.rs` (Terminal trait/impls) from the original codebase.
  - Added `crossterm` to `[patch.crates-io]` in root `Cargo.toml` to ensure the workspace uses our vendored version (which exposes `mod event` unconditionally, unlike the crates.io version which requires the `events` feature).
  - Verified all 90 tests in `tau-tui` pass.
- Files changed:
  - `Cargo.toml` (added `[patch.crates-io]` for crossterm)
  - `crates/tau-tui/Cargo.toml`
  - `crates/tau-tui/src/lib.rs`
  - `crates/tau-tui/src/utils.rs`
  - `crates/tau-tui/src/component.rs`
  - `crates/tau-tui/src/terminal.rs`
- Learnings for future iterations:
  - When using vendored crates that modify API availability (e.g. enabling modules without features), `[patch.crates-io]` is required in the workspace root so that workspace members see the vendored code instead of the registry version. Relying on `dependencies` path is not enough if you want to use the version number, and `patches.list` only affects plugin compilation.
